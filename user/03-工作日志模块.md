# 03 - 工作日志模块

## 模块概述

工作日志模块提供基于 TipTap 的富文本编辑器，支持 Notion 风格的编辑体验，用于记录每日工作内容。

## 主要功能

### 1. 富文本编辑
- **Notion 风格编辑器**：类似 Notion 的编辑体验
- **丰富的格式支持**：
  - 标题（H1-H6）
  - 粗体、斜体、下划线
  - 代码块（支持语法高亮）
  - 任务列表（待办事项）
  - 链接、高亮
  - 文本颜色
  - 文本对齐

### 2. 自动保存
- 编辑内容自动保存为草稿
- 每个用户每天一条草稿记录
- 防止意外关闭导致内容丢失

### 3. 日志管理
- 按日期查看历史日志
- 支持搜索和筛选
- 导出日志内容

### 4. 一日一志
- 每个用户每天只能有一条工作日志
- 当天可多次编辑
- 次日自动创建新日志

## 使用示例

### 编辑器初始化
```vue
<template>
  <NotionEditor
    v-model="content"
    @update="handleUpdate"
    placeholder="开始记录今天的工作..."
  />
</template>

<script setup lang="ts">
import { ref } from 'vue'
import NotionEditor from '@/components/worklog/NotionEditor.vue'

const content = ref({
  type: 'doc',
  content: []
})

const handleUpdate = (newContent) => {
  // 自动保存草稿
  saveDraft(newContent)
}
</script>
```

### 保存日志
```typescript
import { api } from '@/utils/api'

// 保存今日工作日志
const saveWorklog = async (content: any) => {
  const today = new Date().toISOString().split('T')[0]

  await api.post('/api/worklogs', {
    log_date: today,
    content: JSON.stringify(content)
  })
}
```

## 特色功能

### 1. Slash 命令
- 输入 `/` 触发命令菜单
- 快速插入标题、列表、代码块等
- 类似 Notion 的交互体验

### 2. Markdown 快捷输入
- 支持 Markdown 语法快捷输入
- 自动转换为富文本格式
- 提升编辑效率

### 3. 拖拽排序
- 支持内容块拖拽排序
- 可视化调整内容结构

### 4. 实时协作（未来）
- 预留多人实时编辑能力
- 支持内容冲突解决

## 注意事项

1. **内容格式**：
   - 日志内容以 JSON 格式存储
   - 保存前确保内容有效性
   - 避免存储过大的图片（建议使用外链）

2. **自动保存策略**：
   - 编辑停止 2 秒后触发自动保存
   - 草稿和正式日志分开存储
   - 草稿不会覆盖已保存的日志

3. **性能优化**：
   - 长文档使用延迟加载
   - 代码高亮使用 Web Worker
   - 避免频繁重新渲染

## v2 结构化项目日志（2026-05 改造）

### 设计目标
- 工程咨询行业：按「项目 + 办理事项」维度结构化记录日志，一日多条
- 可上传图片 / Word / PDF 附件
- 自动生成甘特图、周报、结题报告
- 通过独立 Linux 服务器上的 OpenAI 兼容大模型提供智能问答与内容生成

### 数据模型（PostgreSQL）

| 表 | 说明 |
|---|---|
| `worklog_districts` | 行政区字典（北京市 16 区） |
| `worklog_project_types` | 项目类型字典（新建项目、公共公益项目） |
| `worklog_matters` | 办理事项字典，含标准办理天数 |
| `worklog_contract_statuses` | 合同付款状态字典 |
| `worklog_projects` | 项目主表：name / client_name / district / project_type / owner_user_id / start_date / is_completed |
| `worklog_entries` | 日志主表：log_date / project_id / matter / user_id / owner_user_id / contract_status / work_note / is_finalized |
| `worklog_progress_notes` | 进展记录表（时间线追加）：entry_id / content / created_by / created_by_name / created_at |
| `worklog_attachments` | 附件表：entry_id / file_kind(image/document) / file_path |
| `worklog_contract_progress` | 合同进度表：project_id / status / amount / note / created_by |
| `worklog_contract_attachments` | 合同附件表：progress_id / file_kind / file_name / file_path |
| `worklog_permissions` | 权限白名单表（周报下载等精细权限） |

> `worklog_entries.is_finalized = true` 后，后端会拒绝任何编辑、删除、加附件操作。

### 权限

| 角色 | 看日志 | 写日志 | 字典管理 | 周报下载 |
|---|---|---|---|---|
| super_admin / admin | 全部 | 全部 | ✓ | 白名单决定 |
| general_manager | 全部 | 全部 | × | 白名单决定 |
| user | 仅自己 | 仅自己 | × | 白名单决定 |
| guest | 仅自己 | × | × | × |

周报下载单独维护在 `worklog_permissions` 表，seed 默认放入姓名为「于贵臣」「刘行」的用户。超级管理员可通过 `/api/worklog-permissions` 动态维护。

### 新增 API

#### 项目 `/api/worklog-projects`
- `GET /` 列表（支持 q / district / ownerUserId / isCompleted 查询）
- `POST /` 新建（同 name+client_name 自动复用已有项目）
- `GET /:id` 详情
- `PUT /:id` 编辑（仅负责人 / admin）
- `POST /:id/complete` 办结项目
- `GET /:id/gantt` 甘特图数据：按办理事项聚合，含超期标记
- `GET /:id/contract` 获取项目合同进度时间线（含附件）
- `POST /:id/contract` 追加合同进度记录（自动更新项目 contract_status 快照）
- `DELETE /:id/contract/:progressId` 删除合同进度记录
- `POST /:id/contract/:progressId/attachments` 上传合同附件
- `DELETE /:id/contract/:progressId/attachments/:attId` 删除合同附件
- `PUT /:id/contract-amount` 更新项目合同总金额

#### 日志 `/api/worklog-entries`
- `GET /` 列表（startDate/endDate/projectId/district/matter/ownerUserId/userId 筛选）
- `POST /` 新建
- `GET /:id` 详情
- `PUT /:id` 更新
- `POST /:id/finalize` 勾选办结（单向）
- `DELETE /:id`
- `POST /:id/attachments` 上传附件（multer，file_kind 必填，图片 ≤10MB、文档 ≤30MB）
- `DELETE /:id/attachments/:attachmentId`

#### 字典 `/api/worklog-dicts`
- `GET /:kind`（kind: districts / project-types / matters / contract-statuses）
- `POST /:kind`、`PUT /:kind/:id`、`DELETE /:kind/:id`（仅 admin，DELETE 为软停用）

#### 报告 `/api/worklog-reports`
- `POST /weekly`：单项目周报（.docx），仅白名单可下载
- `POST /weekly/batch`：批量周报 ZIP
- `POST /summary`：项目结题报告（.docx），负责人 + admin 可生成

#### AI `/api/worklog-ai`
- `GET /status` 返回 LLM 是否已配置
- `POST /chat`：项目问答（可指定 projectId 带上下文）
- `POST /analyze-risk/:projectId`：大模型风险分析（返回 JSON 数组）
- `GET /gantt-alerts`：规则预警（不用大模型；超期 & 临近超期）

#### 权限 `/api/worklog-permissions`
- `GET /me` 查询当前人拥有的精细权限（前端据此显隐按钮）
- `GET /`、`POST /`、`DELETE /:id`（仅超级管理员）

#### 文件下载
- `GET /api/files/worklog/:attachmentId` 按附件 id 拉取文件（带权限校验）

### AI 智能内容补全

在每日工作日志的 TipTap 编辑器中，输入关键词后自动触发 AI 补全：

- **触发条件**：用户停止输入 800ms 后，当前段落文本 >= 4 个字符
- **交互方式**：光标下方弹出浮动面板，展示 3~5 条规范工作描述建议
- **键盘操作**：↑↓ 选择、Enter/Tab 插入、Esc 关闭
- **插入行为**：选中建议后替换当前段落内容
- **降级策略**：LLM 未配置或不可达时，面板不显示，无报错

### 团队日志管理（总经理/管理员）

总经理和系统管理员可查看所有员工的每日日志提交情况：

- **本周概览**：顶部展示周一~周日每天的提交率进度条，点击切换日期
- **日志列表**：选中日期的所有员工日志卡片，HTML 渲染内容
- **评论功能**：总经理/管理员可对员工日志添加评论
- **提交率统计**：右侧面板显示当日提交率百分比
- **未提交人员**：红色高亮显示当日未提交日志的员工

权限控制：
- 路由 `requiresRole: ['super_admin', 'admin', 'general_manager']`
- 后端接口使用 `requireAdminOrGM` 中间件
- 侧边栏菜单对总经理和管理员可见

相关文件：

- `daily_log_comments`（id, submission_id, user_id, content, created_at, read_at, reply_to）— 日志评论

### 评论通知机制

总经理评论员工日志后，员工端会收到通知：

- **侧边栏徽章**：「今日日志」菜单项显示未读评论数量（红色数字）
- **历史日志评论展示**：员工展开历史日志时，每条提交下方显示评论列表（评论人 + 内容 + 时间）
- **自动标记已读**：员工展开含评论的日期时，自动调用 mark-read 接口标记为已读
- **轮询更新**：通过 `pendingStore` 每 10 秒轮询 `/api/approval/pending-counts` 获取未读数

### 评论回复互动

采用扁平回复模式（类似微信朋友圈评论），所有回复平铺显示，回复时标注「回复 @某某」：

- **员工回复**：员工在历史日志评论区点击「回复」按钮，输入回复内容
- **总经理回复**：总经理在团队日志页面评论区点击「回复」按钮
- **回复展示**：带有 replyTo 的评论显示「回复 @某某」标签
- **权限控制**：员工只能回复自己日志下的评论，管理员/总经理可回复任何评论

相关接口：
- `GET /api/daily-logs/history` — 返回每条 submission 的 comments 列表（含 replyTo、replyToUserName）
- `POST /api/daily-logs/comments/:submissionId/reply` — 员工回复评论（requireAuth，校验 submission 归属）
- `POST /api/daily-logs/comments/:commentId/withdraw` — 撤回自己的评论（仅评论作者可操作）
- `POST /api/daily-logs/team/comments/:submissionId` — 总经理评论/回复（支持 replyTo 参数）
- `POST /api/daily-logs/comments/mark-read` — 标记评论已读（入参 submissionIds）
- `GET /api/approval/pending-counts` — 返回 unreadLogComments 字段
